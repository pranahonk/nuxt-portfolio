## Problem Statement

The current `/api/posts` endpoint is a performance bottleneck due to fetching all public Notion posts on every request without caching. This results in significant latency (~5-10s for 2300 posts) and large payloads (~30KB minified). The frontend component `pages/posts/index.vue` implements "fake pagination" by downloading all posts and slicing them client-side, negating any network benefits. Back-navigation is slow, as Nuxt unmounts the page component and triggers a full refetch. There is also no service worker for offline support or asset caching.

## Goals

- **Reduce API response time:** Implement server-side caching and pagination for `/api/posts`.
- **Improve frontend performance:** Implement true pagination and lazy loading.
- **Enhance user experience:** Provide instant back-navigation and offline capabilities.
- **Maintain freshness:** Ensure new posts appear rapidly without full page reloads.

## Proposed Architecture

A hybrid approach combining static generation, Netlify Blob caching, and Nuxt's `KeepAlive` with a service worker.

### Server-Side (`/api/posts`)

1.  **Primary Data Source:** `public/api/posts.json` (static file). This file will be generated by an updated `scripts/generate-posts-json.js` on every news-sync cron run. It will contain all public Notion posts, sorted by `created_at` descending.
2.  **Fast Cache:** Netlify Blob Storage. A single key (`posts:listing`) will store the full sorted post list. This cache will have a short TTL (e.g., 5 minutes) to ensure freshness between cron runs.
3.  **Cold Path / Source of Truth:** Notion API. This will only be hit if the Blob cache is expired/missing *and* the static `posts.json` is not sufficient or out of date.

The `/api/posts` endpoint will implement **Offset Pagination**, returning a slice of posts based on `page` and `limit` query parameters, along with `total` and `hasMore` metadata.

### Client-Side (`pages/posts/index.vue`)

1.  **True Pagination:** The component will fetch posts page by page using the `/api/posts?page={N}&limit=10` endpoint.
2.  **Infinite Scroll:** The existing `useIntersectionObserver` will trigger fetching of the next page.
3.  **Instant Back-Navigation:** The `pages/posts/index.vue` component will be wrapped in `<KeepAlive>` to preserve its state (including scroll position and fetched data) when navigating to individual posts and returning.
4.  **Offline Support:** A service worker will be implemented using `@vite-pwa/nuxt` to provide offline capabilities and asset caching.

## Detailed Design

### 1. `server/api/posts/index.ts` (API Endpoint)

This endpoint will be responsible for serving paginated post data.

-   **Input:** `page` (number, default 1), `limit` (number, default 10) query parameters.
-   **Output:** `{ posts: Article[], total: number, hasMore: boolean }`.
-   **Logic:**
    1.  Attempt to fetch the full post list from Netlify Blob storage (`posts:listing` key). If found and not expired, use it.
    2.  If Blob cache is missing/expired, attempt to read `public/api/posts.json`.
    3.  If both fail or are deemed outdated (e.g., if the blob cache miss triggers a refresh), query Notion API for all public posts.
    4.  Store the full fetched list in Blob storage (`posts:listing`) with a 5-minute TTL.
    5.  Apply pagination: `posts.slice((page - 1) * limit, page * limit)`.
    6.  Return the paginated slice, total count, and `hasMore` flag.

### 2. `server/utils/post-store.ts` (Blob Storage Helpers)

New functions will be added:

-   `getPostListing(): Promise<Article[] | null>`: Fetches the full post list from Blob storage.
-   `setPostListing(posts: Article[], ttl: number): Promise<void>`: Stores the full post list in Blob storage with a specified TTL.

### 3. `scripts/generate-posts-json.js` (Static File Generator)

This script will be updated to:

1.  Query Notion API for all public posts (using the same auth pattern as `/api/posts` cold path).
2.  Merge these with local JSON articles from `server/data/articles/` (Notion data takes precedence).
3.  Sort the combined list by `created_at` descending.
4.  Write the full sorted list to `public/api/posts.json`.
5.  Call `setPostListing(posts, TTL_LONG)` to update the Netlify Blob cache for `posts:listing`, ensuring a fresh canonical source.
6.  This script will be integrated into the news-sync GitHub Actions workflow, running after the `enrich` step.

### 4. `pages/posts/index.vue` (Frontend Component)

-   Modify `useLazyFetch` call to include `page` and `limit` parameters:
    ```vue
    const page = ref(1);
    const limit = PAGE_SIZE; // defined as 10
    const { data, pending, error } = useLazyFetch(() => `/api/posts?page=${page.value}&limit=${limit}`);

    // In loadMore function:
    page.value++;
    ```
-   Update `allPosts` computed property to accumulate posts from successive pages.
-   Wrap the `<template>` content with `<KeepAlive>` to preserve state on back-navigation.

### 5. `nuxt.config.ts` (Nuxt Configuration)

-   Add `@vite-pwa/nuxt` module.
-   Configure the PWA module for service worker generation with a **Network-first** strategy for API routes and **Cache-first** for static assets.
    ```typescript
    pwa: {
      workbox: {
        navigateFallback: '/index.html',
        globPatterns: ['**/*.{js,css,html,png,svg,webp,woff2}'],
      },
      client: {
        installPrompt: true,
      },
      manifest: {
        name: 'Prana Wijaya',
        short_name: 'Prana',
        theme_color: '#1a1a2e',
      },
    }
    ```

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Blob store unavailable in local dev | Certain | Existing `post-store.ts` already handles this gracefully (degrades to static `posts.json`). |
| `@vite-pwa` conflicts with existing CSP headers | Low | Audit `netlify.toml` headers after adding `@vite-pwa/nuxt`; adjust if needed. |
| Cron run is skipped â†’ static list grows stale | Low | GitHub Actions notification on cron failure; static list still serves, just older content. Blob cache provides a dynamic refresh layer. |
| Offset pagination + concurrent Notion insert = off-by-one | Negligible | Notion list is append-only by date; any discrepancy is minimal and self-corrects on next request. |

## Acceptance Criteria

-   `/api/posts` responds within milliseconds (after initial cold start/cache fill).
-   `/api/posts?page=X&limit=Y` returns correct paginated data.
-   `pages/posts/index.vue` loads posts incrementally via true pagination.
-   Navigating `posts/index.vue` â†’ `posts/[slug].vue` â†’ `posts/index.vue` is instant (scroll position preserved, no visible refetch).
-   Site provides basic offline experience via service worker.
